The Shift

From conductor
to orchestrator

Addy Osmani's framework for how AI coding is evolving.


🎵 Conductor

One agent, synchronous. You guide it line by line. Your context window is the ceiling. This is where most people are today.

🎼 Orchestrator

Multiple agents, asynchronous. You plan and verify. Each agent has its own context, scope, responsibility. The codebase is your canvas.

Source: Addy Osmani — "The Code Agent Orchestra" (O'Reilly AI CodeCon, March 2026)

Addy Osmani gave a great talk at O'Reilly last week about this. He describes the shift from being a conductor — where you guide a single AI in real time — to being an orchestrator, where you manage a team of agents working asynchronously. Most developers today are conductors. The best ones are becoming orchestrators. But here's what nobody's talking about yet: there's a level BEYOND orchestrating code agents. And that's orchestrating infrastructure agents.

The Serious Part

"But isn't this
dangerous?"

Let's talk about what can go wrong.

Now the question every responsible engineer is thinking. "You're giving an AI agent access to production infrastructure? Are you insane?" Fair question. Let's be honest about the risks.

What can go wrong?


Misinterpretation

"Delete the test environment" → deletes production. Ambiguous commands are the #1 risk.

Over-provisioning

Agent spins up D64s_v5 when B2ms would do. LLMs default to "more" unless constrained.

Data access

Agent with broad RBAC could read Key Vault secrets, customer data, logs it shouldn't see.

Runaway loops

Agent retries a failed deployment 47 times. Each retry creates resources. Bill goes nuclear.


These are real risks. Not theoretical. Let's talk about what's in our control.

I'm not going to sugarcoat this. An agent can misinterpret commands. It can over-provision. It can access data it shouldn't. It can get stuck in loops and burn through your budget. These are real risks. But — and this is the key insight — Azure already has the governance stack to handle this.

What's in our control


🔐 Azure AD + RBAC

Least privilege. Scoped tokens. The agent can only touch what you explicitly allow. Time-limited, revocable.

⚖️ Azure Policy

Deny by default. Agent can't create non-compliant resources even if it tries. Enforced at the platform level.

🔒 Resource Locks

CanNotDelete on production. Agent physically cannot destroy what's locked. No override without human.


🚧 Human-in-the-Loop

Approval gates for destructive ops. Agent asks before it deletes. No YOLO mode.

📝 Activity Log

Every action traced. Full audit trail. Who asked, what the agent did, when, and why.

🧱 Agent Sandboxing

Deny lists, command allowlists, resource group boundaries. The agent operates in a box.

Here's what IS in our control. Azure AD and RBAC for least privilege. Azure Policy — deny by default, the agent literally cannot create non-compliant resources. Resource Locks — CanNotDelete on anything important. Human approval gates for destructive operations. Full audit trail in Activity Log. And agent-level sandboxing — deny lists, allowlists, boundaries. This is Azure's governance stack. It was built for humans. And here's the thing — it works even better for agents, because agents actually follow the rules.

What's NOT yet in our control


  • Hallucination in IaC generation — Agent might generate Bicep that compiles but does the wrong thing
  • Token/credential scope management — Still largely manual. No standard tooling yet
  • Agent-to-agent trust — No standard for how agents delegate to each other safely
  • Prompt injection in multi-user scenarios — Shared agent surfaces are an attack vector
  • Cost ceiling enforcement — Azure Budgets alert but don't hard-stop a runaway agent


This is an evolving space. Governance patterns are still forming.
That's exactly why your customers need you.

Let me be honest about what we DON'T control yet. Hallucination in IaC — an agent can generate Bicep that compiles but provisions the wrong thing. Credential scope is still manual. Governance patterns are still forming. And that's exactly why your customers need you — to help them navigate this responsibly.

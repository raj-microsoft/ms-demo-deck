Generated Bicep — main.bicep


// Auto-generated by OpenClaw from conversation
// Customer: Freelancer SaaS platform

@description('Azure region for all resources')
param location string = resourceGroup().location

@description('App name prefix')
param appName string = 'freelancer-saas'

// App Service Plan
resource appServicePlan 'Microsoft.Web/serverfarms@2023-12-01' = {
  name: '${appName}-plan'
  location: location
  sku: {
    name: 'B1'
    tier: 'Basic'
  }
  kind: 'linux'
  properties: {
    reserved: true
  }
}

// App Service — Next.js
resource webApp 'Microsoft.Web/sites@2023-12-01' = {
  name: '${appName}-web'
  location: location
  properties: {
    serverFarmId: appServicePlan.id
    siteConfig: {
      linuxFxVersion: 'NODE|20-lts'
      appSettings: [
        { name: 'DATABASE_URL', value: sqlServer.properties.fullyQualifiedDomainName }
        { name: 'AZURE_STORAGE_CONNECTION', value: storageAccount.properties.primaryEndpoints.blob }
      ]
    }
  }
}
This is the Bicep that OpenClaw generates. Notice — it's not template code. It's specific to this customer's requirements. App Service Plan on Linux, Node 20, with the right app settings wired up to the SQL server and storage account. Let me show you the database part.

What OpenClaw ran under the hood


# 1. Detected via Application Insights alert
az monitor app-insights query --app freelancer-saas-insights \
  --analytics-query "requests | where resultCode >= 500 | count"

# 2. Pulled recent config changes
az webapp config appsettings list \
  --name freelancer-saas-web --resource-group rg-freelancer-saas

# 3. Compared with deployment history
az webapp deployment list --name freelancer-saas-web \
  --resource-group rg-freelancer-saas

# 4. Identified root cause: DATABASE_URL changed at 02:48 UTC
# 5. Rolled back config
az webapp config appsettings set \
  --name freelancer-saas-web \
  --resource-group rg-freelancer-saas \
  --settings DATABASE_URL="Server=freelancer-saas-sql.database..."

# 6. Restarted
az webapp restart --name freelancer-saas-web \
  --resource-group rg-freelancer-saas

# 7. Verified health
curl -s -o /dev/null -w "%{http_code}" \
  https://freelancer-saas-web.azurewebsites.net/api/health

The elephant in the room

"But isn't giving an AI
cloud access dangerous?"

Security. Governance. Guardrails. Traceability.

Now — the question every CISO, every compliance officer, every responsible engineer is thinking. "You're giving an AI agent access to production infrastructure? Are you insane?" Fair question. Let me show you why Azure makes this safe.
🛡️

Security by design


🔐 Least Privilege

Agents get scoped tokens, not admin keys. Managed Identity + RBAC = the agent can only touch what you allow. Time-limited, revocable.

🚧 Guardrails

Destructive actions require human approval. Delete a resource group? Agent asks first. Deploy to production? Gated. No YOLO mode.

🧱 Blast Radius

Sandboxed execution. Agents operate in defined resource groups, subscriptions, and policy boundaries. Azure Policy enforces what code can't.

Three layers of security. First — least privilege. The agent doesn't get admin access. It gets a Managed Identity with specific RBAC roles. It can deploy to dev but not prod. It can read costs but not delete subscriptions. Second — guardrails. Destructive actions require human approval. The agent ASKS before it deletes. Third — blast radius. Azure Policy, resource locks, and subscription boundaries contain the agent. Even if it goes rogue, damage is limited.
📋

Governance & Traceability


📝 Full Audit Trail

Every action logged. Azure Activity Log + agent session transcripts = complete chain of custody. Who asked, what the agent did, when, and why.

⚖️ Azure Policy

Enforce compliance at the platform level. Agent can't create resources that violate policy — even if it tries. Deny by default.

🔍 Explainability

Agent explains its reasoning before acting. "I'm going to scale down these VMs because..." Full transparency. No black boxes.

Governance. Every single action the agent takes is logged in Azure Activity Log AND in the agent's own session transcript. Full traceability — who asked it, what it did, what reasoning it used. Azure Policy enforces compliance at the platform level — the agent literally cannot create non-compliant resources. And explainability — the agent tells you WHY it's doing something before it does it. No black boxes.
